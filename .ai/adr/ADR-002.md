# ADR-002: Separación de la configuración por dominio y eliminación del acoplamiento global de getAppConfig()

## 1. Estado

Estado de la decisión: **Aceptado**

Estado de implementación: **Parcial**

La decisión arquitectónica está formalmente aprobada y constituye arquitectura vigente dentro de su alcance. La aceptación autoriza la migración incremental definida en este ADR, pero no declara completada su implementación ni sustituye autorizaciones específicas para cada cambio operativo.

Existe una primera implementación verificable para Auth. La compatibilidad legacy y las etapas restantes continúan abiertas.

## 2. Fecha

- Propuesta documentada: **2026-07-22**.
- Aceptación formal: **2026-07-22**.

## 3. Contexto

La implementación actual centraliza la lectura, validación y caché de configuración en `libs/config`.

El flujo actual de autenticación es:

```text
JwtStrategy
  -> getAuthConfig()
  -> loadAuthConfig()
  -> validateAuthEnvironment()
  -> AZURE_TENANT_ID + AZURE_CLIENT_ID
```

`JwtStrategy` ya no consume `getAppConfig()`: utiliza el módulo Auth independiente y valida únicamente la configuración necesaria para Microsoft Entra ID.

La API legacy continúa disponible temporalmente mediante este flujo:

```text
getAppConfig()
  -> validateRequiredEnvironmentVariables()
  -> loadAppConfig()
  -> NexusConfig global
```

`getAppConfig()` conserva una configuración global que incluye variables relacionadas con:

- Azure Key Vault.
- Azure Storage.
- Dataverse.
- Business Central.
- Configuración general de aplicación.

Algunas de estas integraciones todavía no están implementadas funcionalmente. Su configuración ya no afecta la inicialización de `JwtStrategy`, pero la API global permanece como superficie de compatibilidad hasta completar el inventario y retiro seguro del código legacy.

El estado descrito es una **Implementación actual parcial** y una **deuda técnica conocida**. El desacoplamiento de Auth está implementado; la configuración global restante no representa la arquitectura deseada.

## 4. Problema

La configuración global introduce los siguientes problemas:

- `JwtStrategy` queda acoplada a dominios e integraciones que no utiliza.
- Cada consumidor recibe potencialmente más configuración de la que necesita.
- La validación all-or-nothing puede bloquear despliegues por variables ajenas al módulo ejecutado.
- Antes de la migración de Auth, cambios en Dataverse, Business Central, Storage o Key Vault podían afectar indirectamente a autenticación; la API legacy conserva ese riesgo para cualquier consumidor que vuelva a utilizarla.
- Un único punto de entrada no establece una frontera técnica entre configuración browser y server.
- Secretos server-side podrían quedar disponibles accidentalmente para consumidores que no deberían conocerlos.
- La presencia de variables puede confundirse con la disponibilidad funcional de una integración.
- Las pruebas aisladas de cada módulo requieren configurar dependencias no relacionadas.

## 5. Decisión aceptada

Se adopta el reemplazo incremental de la configuración global obtenida mediante `getAppConfig()` por contratos, cargadores y validadores independientes por dominio o módulo.

Cada módulo consumiría únicamente su propia configuración y validaría solo las variables que utiliza. La configuración pública de frontend, la configuración server-side y los secretos se expondrían mediante fronteras y puntos de entrada distintos.

La migración eliminará `getAppConfig()` después de que todos sus consumidores hayan sido trasladados y validados. No se realizará una eliminación inmediata que rompa consumidores existentes.

Esta decisión está **Aceptada**. Su implementación debe seguir el plan incremental y las fronteras definidas en este ADR, sin ampliar el alcance a módulos que todavía no tengan un consumidor real aprobado.

## 6. Arquitectura aceptada

### 6.1 Configuración de Auth

Estado: **Implementación actual parcial y verificable en el working tree**.

- Contrato mínimo para Microsoft Entra ID y validación JWT.
- Cargador y validador propios.
- Acceso únicamente a tenant, client ID o audience y demás valores que autenticación necesite realmente.
- Sin dependencia de Dataverse, Business Central, Azure Storage o Azure Key Vault salvo necesidad real, explícita y documentada.
- Servicio con caché propia y punto de entrada `@nexus/config/auth`.
- `JwtStrategy` consume `getAuthConfig()` y mantiene `clientId` como audience.

### 6.2 Configuración de Dataverse

- Contrato, loader y validador independientes.
- Validación ejecutada únicamente cuando se inicialice el adaptador de Dataverse.
- Sin exposición de su configuración al frontend o a módulos no consumidores.

### 6.3 Configuración de Business Central

- Contrato, loader y validador independientes.
- Validación limitada a las variables requeridas por el adaptador de Business Central.
- Separación respecto a Auth y a otros proveedores.

### 6.4 Configuración de Azure Storage

- Contrato, loader y validador independientes.
- Inicialización únicamente cuando exista un consumidor funcional de Storage.
- Acceso server-side y con el mínimo conjunto de valores necesario.

### 6.5 Configuración de Azure Key Vault

- Contrato y proveedor server-side independientes.
- Sin acceso desde browser ni desde módulos que no administren secretos.
- La estrategia concreta de identidad, caché y disponibilidad se definirá antes de implementar el proveedor.

### 6.6 Separación Browser / Server

- Punto de entrada público específico para configuración segura de browser.
- Punto de entrada público específico para configuración server-side.
- Los secretos y loaders basados en `process.env` no se exportan desde el punto de entrada browser.
- Las dependencias de browser hacia configuración server-side deben impedirse mediante límites verificables.
- Los tipos compartidos no deben inducir a que secretos formen parte de contratos de cliente.

### 6.7 Validación por módulo

- Cada módulo valida únicamente sus variables.
- La ausencia de configuración de una integración no utilizada no bloquea otros módulos.
- Los errores identifican el módulo y la variable faltante sin exponer valores sensibles.
- La caché, si se conserva, opera por contrato o loader y no mediante un objeto global único.

## 7. Beneficios esperados

- Menor acoplamiento entre autenticación e integraciones empresariales.
- Despliegues que fallan únicamente por configuración requerida por las capacidades activas.
- Pruebas unitarias y de integración más aisladas.
- Contratos pequeños y explícitos.
- Menor disponibilidad accidental de secretos dentro del proceso.
- Frontera verificable entre browser y server.
- Incorporación incremental de integraciones todavía no implementadas.
- Mayor claridad sobre qué módulo es propietario de cada variable.
- Menor impacto transversal al modificar una integración.
- Mejor alineación con mínimo privilegio y responsabilidad única.

## 8. Riesgos

- La migración puede romper consumidores existentes si se elimina `getAppConfig()` prematuramente.
- Durante la transición pueden coexistir configuración global y loaders específicos.
- La duplicación temporal de lectura o validación puede generar inconsistencias.
- Los nombres y defaults actuales pueden requerir compatibilidad transitoria.
- Un punto de entrada browser separado no es suficiente si los límites de importación no se validan.
- La fragmentación excesiva puede crear loaders demasiado pequeños o difíciles de descubrir.
- Distintos caches de configuración pueden comportarse de forma inconsistente si no se define una política común.
- Los errores de configuración podrían cambiar de formato y afectar pruebas u operación.

Mitigaciones propuestas:

- Migración por consumidor.
- Compatibilidad temporal controlada.
- Tests de contratos y fallos de configuración.
- Revisión de imports browser/server.
- Eliminación de la API global únicamente al finalizar la migración.

## 9. Plan de migración incremental

La aceptación de este ADR autoriza la ejecución incremental de este plan. Cada etapa debe conservar compatibilidad, cumplir sus validaciones y respetar las autorizaciones específicas aplicables.

### Etapa 1. Inventario

Estado: **Completada para el working tree inspeccionado**.

- Se identificó a `JwtStrategy` como consumidor activo de `getAppConfig()` antes de la migración.
- Se confirmó que las configuraciones de integraciones restantes no tienen consumidores funcionales implementados.
- El inventario deberá repetirse antes de eliminar la API legacy para detectar consumidores incorporados posteriormente.

### Etapa 2. Contratos independientes

Estado: **Parcialmente completada**.

- Se implementó el contrato mínimo de Auth.
- Se creó `.ai/config-module-standard.md` como estándar técnico para módulos futuros.
- Permanecen pendientes los contratos de otros módulos, que solo deberán crearse con un consumidor real aprobado.
- La separación efectiva entre configuración browser, server-side y secretos permanece pendiente.

### Etapa 3. Migración de Auth

Estado: **Completada**.

- Se crearon contrato, loader, validador, servicio, error y pruebas de Auth.
- `JwtStrategy` depende exclusivamente de `getAuthConfig()`.
- Las pruebas verifican que Auth inicia sin variables de Dataverse, Business Central, Azure Storage ni Azure Key Vault.

### Etapa 4. Migración de integraciones

Estado: **Pendiente**.

- Crear cada loader solo junto con su consumidor funcional o necesidad aprobada.
- Migrar consumidores existentes de forma individual.
- Evitar crear adaptadores funcionales fuera del alcance de la configuración.
- Mantener pendientes Observability, Storage, Key Vault, Dataverse y Business Central hasta que cada módulo tenga alcance y consumidor autorizados.

### Etapa 5. Puntos de entrada Browser / Server

Estado: **Pendiente**. Solo existe el subpath server-side específico `@nexus/config/auth`; no existe todavía una frontera Browser/Server completa.

- Publicar puntos de entrada separados.
- Migrar imports existentes.
- Añadir controles para impedir imports server-side desde aplicaciones browser.

### Etapa 6. Retiro de la configuración global

Estado: **Pendiente**.

- Confirmar que no quedan consumidores de `getAppConfig()`.
- Retirar la validación global y el objeto `NexusConfig` global.
- Eliminar los exports legacy únicamente después de validar todos los consumidores.
- Actualizar documentación y pruebas en el mismo cambio.

## 10. Criterios de finalización de la implementación

La decisión ya está aceptada. Su implementación se considerará completada únicamente cuando:

1. `JwtStrategy` dependa únicamente de la configuración requerida para Auth.
2. Auth pueda inicializarse sin variables de Dataverse, Business Central, Azure Storage o Azure Key Vault cuando esas integraciones no estén activas.
3. Existan cargadores y validadores independientes para cada módulo que tenga un consumidor real.
4. Cada módulo falle únicamente por sus propias variables ausentes o inválidas.
5. La configuración pública de frontend esté separada de configuración server-side y secretos.
6. El punto de entrada browser no exporte secretos, loaders server-side ni acceso genérico a `process.env`.
7. Los límites de importación browser/server sean verificables.
8. No existan consumidores de `getAppConfig()`.
9. La API global de configuración haya sido retirada de manera compatible.
10. Los tests unitarios de loaders, Auth y errores de configuración sean satisfactorios.
11. Build, lint y tests de los proyectos afectados sean satisfactorios.
12. La documentación refleje la implementación final.

## Métricas de éxito

La implementación de la decisión se evaluará mediante los siguientes indicadores verificables:

- **Cumplida:** `JwtStrategy` inicia únicamente con configuración de Auth.
- **Cumplida para Auth:** la autenticación puede iniciar sin variables de módulos no utilizados.
- **Cumplida para Auth:** su loader valida únicamente sus propias variables.
- **Pendiente:** separación efectiva entre Browser y Server.
- **Pendiente:** eliminación completa de `getAppConfig()`.
- **Pendiente:** eliminación completa del `NexusConfig` global.
- **Cumplida para el módulo implementado:** no existen dependencias cruzadas desde Auth hacia otros loaders.
- **Cumplida para el primer corte:** build, lint y tests de `config` y `api` fueron satisfactorios.
- **Cumplida para el primer corte:** documentación relacionada actualizada.

## 11. Plan de reversión

La migración aceptada debe ser reversible por etapas.

- Mantener temporalmente `getAppConfig()` mientras se migra cada consumidor.
- No eliminar exports globales hasta confirmar que no existen dependientes.
- Si un loader independiente introduce una regresión, restaurar únicamente el consumidor afectado al adaptador de compatibilidad.
- Conservar nombres de variables durante la transición para evitar cambios simultáneos de configuración e implementación.
- Revertir una etapa antes de continuar si sus pruebas o validaciones fallan.
- No usar una reversión destructiva de Git ni descartar cambios ajenos.

Si esta decisión se reemplaza en el futuro, el nuevo ADR debe indicar expresamente que sustituye a ADR-002 y definir la transición correspondiente.
