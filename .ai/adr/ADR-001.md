# ADR-001: Adoptar Nx Monorepo para las aplicaciones y librerías de NEXUS

## 1. Estado

**Aceptado**

La decisión está materializada en el código versionado y constituye **Arquitectura vigente** de NEXUS.

El estado **Aceptado** aprueba únicamente la adopción de Nx Monorepo y la estructura principal definida en la sección 6. No aprueba automáticamente evoluciones posteriores descritas como arquitectura propuesta o implementación futura.

## 2. Fecha

- Decisión implementada inicialmente: **2026-07-17**.
- Librería compartida inicial incorporada: **2026-07-20**.
- Formalización de este ADR: **2026-07-22**.

Las dos primeras fechas se basan en el historial Git de `nx.json`, las aplicaciones y `libs/config`.

## 3. Título de la decisión

Adoptar un Nx Monorepo como estructura oficial de NEXUS, compuesto inicialmente por:

- `customer-portal`.
- `admin-portal`.
- `api`.
- Proyectos de pruebas E2E asociados.
- Librerías compartidas bajo `libs/`, comenzando por `libs/config`.

## 4. Contexto

NEXUS combina dos experiencias web, un backend y capacidades compartidas dentro de una misma plataforma empresarial:

- Frontends con Next.js App Router.
- Backend con NestJS.
- Tipos, configuración y utilidades potencialmente reutilizables.
- Pruebas unitarias y E2E.
- Integraciones previstas con Microsoft Entra ID, Azure, Dataverse, Business Central y SharePoint.

Estas piezas necesitan evolucionar de forma coordinada, mantener contratos compatibles y compartir estándares de build, lint, pruebas y TypeScript. El repositorio fue creado como un workspace Nx y posteriormente incorporó una primera librería compartida.

## 5. Problema

NEXUS necesita una estructura que permita:

- Mantener varias aplicaciones relacionadas sin duplicar configuración y tooling.
- Compartir código mediante límites explícitos y puntos de entrada públicos.
- Visualizar y controlar dependencias entre proyectos.
- Validar únicamente proyectos afectados cuando sea apropiado.
- Coordinar cambios de contratos entre frontends, API y librerías.
- Escalar modularmente sin dividir prematuramente el sistema en repositorios o servicios independientes.

Sin una estructura común, las aplicaciones podrían divergir en versiones, estándares, contratos, seguridad y procesos de validación.

## 6. Decisión

NEXUS se mantiene como un **Nx Monorepo** administrado con pnpm y TypeScript.

Las unidades desplegables iniciales son:

- `customer-portal`: aplicación Next.js App Router para la experiencia de clientes.
- `admin-portal`: aplicación Next.js App Router para la experiencia administrativa.
- `api`: aplicación NestJS que actúa como límite backend.

La estructura admite librerías compartidas bajo `libs/`; `libs/config` es la primera librería compartida existente. Las reglas detalladas sobre cohesión, composition roots y dependencias permitidas pertenecen a la arquitectura propuesta y no quedan aprobadas por este ADR.

Nx es responsable de representar el grafo de proyectos, inferir u orquestar targets y permitir validaciones específicas o `affected`.

El alcance formal de aprobación de este ADR se limita a:

- El uso de Nx Monorepo.
- La existencia separada de `customer-portal`, `admin-portal` y `api`.
- Los proyectos de pruebas asociados.
- La capacidad de incorporar librerías compartidas bajo `libs/`, representada actualmente por `libs/config`.

Tags Nx específicos, límites estrictos y cualquier nueva estructura de capas son propuestas no aprobadas por este ADR. La separación Browser/Server de configuración está aprobada de forma independiente por ADR-002 y permanece en implementación parcial.

## 7. Implementación actual

Esta sección toma como referencia el código versionado en `HEAD`, respaldado por el historial Git indicado en la sección 2. No depende de cambios no confirmados del working tree.

### Implementación actual

- Workspace Nx configurado mediante `nx.json`.
- Gestión de dependencias con pnpm.
- Aplicaciones `customer-portal`, `admin-portal` y `api`.
- Proyectos `customer-portal-e2e`, `admin-portal-e2e` y `api-e2e`.
- Librería `libs/config` con punto de entrada `@nexus/config`.
- Plugins Nx para Next.js, Jest, Playwright, ESLint y Webpack.
- CI básico para lint, tests y build.

### Implementación actual incompleta

- La regla de límites Nx existe, pero los proyectos todavía tienen `tags: []`.
- Solo existe una librería compartida.
- La separación entre código browser y server dentro de configuración todavía debe reforzarse.
- Las aplicaciones conservan en gran medida scaffolding inicial.

## 8. Arquitectura propuesta

El monorepo debe evolucionar gradualmente hacia:

- Aplicaciones pequeñas enfocadas en composición y presentación.
- Librerías organizadas por capacidad y capa cuando existan casos de uso reales.
- Separación explícita entre dominio, aplicación, infraestructura y presentación.
- Tags Nx por tipo, scope y plataforma.
- Restricciones automatizadas de dependencias.
- Puntos de entrada públicos mínimos.
- Separación estricta entre librerías browser, server e isomórficas.
- Contratos compartidos neutrales respecto a frameworks cuando sean necesarios.

Esta arquitectura propuesta no forma parte de la aprobación de ADR-001. Cada decisión estructural que exceda el alcance de la sección 6 requiere análisis y, cuando corresponda, un ADR propio con estado **Aceptado**. La propuesta no autoriza la creación anticipada de librerías o capas vacías.

## 9. Alternativas consideradas

El historial disponible no contiene una evaluación formal de alternativas anterior a la creación del workspace. Para dejar trazabilidad sin atribuir deliberaciones no registradas, se documentan las alternativas estructurales directamente relacionadas con la decisión existente.

### 9.1 Repositorios independientes por aplicación

No adoptado.

Implicaría separar `customer-portal`, `admin-portal`, `api` y librerías. Aumentaría la autonomía de releases, pero también la duplicación de tooling, coordinación de contratos y mantenimiento de dependencias compartidas.

### 9.2 Monorepo sin Nx

No adoptado.

Mantendría el código en un solo repositorio, pero requeriría construir o mantener mecanismos adicionales para el grafo de dependencias, targets, caché y análisis de proyectos afectados.

### 9.3 Una única aplicación frontend

No adoptado.

Reduciría el número de aplicaciones, pero mezclaría experiencias con audiencias, navegación y posibles ciclos de evolución distintos. El repositorio ya materializa dos portales separados.

### 9.4 Microservicios y repositorios distribuidos desde el inicio

No adoptado.

Introduciría coordinación, despliegue y operación distribuidos sin evidencia actual que justifique ese costo. NEXUS mantiene una API modular como punto de partida.

## 10. Consecuencias positivas

- Configuración y tooling centralizados.
- Versiones coordinadas de TypeScript, Nx y frameworks.
- Cambios atómicos entre aplicaciones, API y contratos compartidos.
- Grafo explícito de dependencias.
- Validaciones específicas o basadas en proyectos afectados.
- Reutilización controlada mediante librerías.
- Menor duplicación de configuración.
- Capacidad de aplicar políticas arquitectónicas con tags y ESLint.
- Pruebas E2E asociadas a cada unidad desplegable.
- Evolución incremental hacia un monolito modular sin distribución prematura.

## 11. Consecuencias negativas

- El repositorio y su pipeline pueden crecer en complejidad con el número de proyectos.
- Una configuración raíz incorrecta puede afectar varios proyectos.
- Las dependencias centralizadas pueden ocultar qué proyecto es propietario de cada paquete.
- Los cambios transversales pueden ampliar el impacto de una modificación.
- La reutilización mal diseñada puede producir librerías genéricas y alto acoplamiento.
- El equipo debe comprender Nx, su grafo, targets y caché.
- Los límites no se aplican automáticamente si tags y restricciones permanecen sin configurar.

## 12. Riesgos

- Dependencias circulares entre librerías.
- Imports profundos que eludan APIs públicas.
- Dependencias directas entre aplicaciones.
- Exposición accidental de código server-side al frontend.
- Crecimiento de una librería `shared` sin responsabilidad clara.
- Acoplamiento de dominio a proveedores externos.
- Ejecución innecesaria de todo el monorepo en CI.
- Divergencia entre la arquitectura documentada y el grafo real.
- Creación anticipada de librerías sin consumidores ni casos de uso.

Mitigaciones:

- Tags y restricciones Nx.
- Revisión periódica del grafo.
- Puntos de entrada públicos mínimos.
- Separación por plataforma.
- Validaciones `affected` y ownership explícito.
- ADR para excepciones o cambios estructurales.

## 13. Impacto arquitectónico

- Define el repositorio como unidad de coordinación técnica.
- Establece aplicaciones como unidades desplegables separadas.
- Establece librerías como unidades de modularidad y reutilización.
- Permite proponer y controlar posteriormente reglas de dependencias mediante las capacidades de Nx.
- Permite adoptar posteriormente Clean Architecture mediante decisiones adicionales.
- Mantiene abierta una evolución futura sin comprometer una división en microservicios.

## 14. Impacto en seguridad

Impactos positivos:

- Permite centralizar reglas, configuración de seguridad y validaciones.
- Facilita contratos y controles consistentes.
- Permite automatizar límites browser/server.

Riesgos de seguridad:

- Una librería compartida mal clasificada puede exponer secretos o código server-side al browser.
- Dependencias amplias pueden expandir la superficie afectada por una vulnerabilidad.
- La configuración central puede convertirse en un punto de exposición si no se separa por plataforma.

Requisito: toda librería debe declarar y respetar su plataforma de ejecución antes de manejar secretos o capacidades privilegiadas.

## 15. Impacto en rendimiento

- Nx puede reducir tiempo de validación mediante caché y ejecución por proyectos afectados.
- Compartir contratos y utilidades evita trabajo duplicado en tiempo de desarrollo.
- El tamaño del monorepo puede aumentar costos de instalación, análisis y CI si los límites no se mantienen.
- Una librería compartida importada incorrectamente puede aumentar bundles frontend.
- El monorepo no determina por sí solo el rendimiento en producción; cada aplicación sigue siendo una unidad desplegable independiente.

Las optimizaciones deben basarse en mediciones y no en la estructura del repositorio por sí sola.

## 16. Impacto en mantenibilidad

Impactos positivos:

- Estándares y actualizaciones coordinados.
- Cambios contractuales trazables en un solo diff.
- Menor duplicación de configuración.
- Descubrimiento de dependencias mediante el grafo Nx.

Costos:

- Requiere disciplina para evitar librerías genéricas.
- Requiere ownership y nomenclatura consistentes.
- Requiere mantener configuración raíz compatible con todos los proyectos.
- Requiere revisar periódicamente límites y dependencias.

## 17. Alcance

Esta decisión cubre:

- Organización del código fuente en un Nx Monorepo.
- Existencia separada de `customer-portal`, `admin-portal` y `api`.
- Proyectos de prueba asociados.
- Creación de librerías compartidas bajo `libs/` cuando estén justificadas.
- Orquestación de build, lint, tests y E2E mediante Nx.

## 18. Elementos fuera de alcance

Esta decisión no define:

- Funcionalidades empresariales de los portales.
- Modelo de dominio o propiedad de datos.
- Contratos específicos de API.
- Implementación de Dataverse, Business Central o SharePoint.
- Topología de despliegue o infraestructura Azure.
- Estrategia detallada de autenticación y autorización.
- Estrategia de releases o versionado de aplicaciones.
- División futura en microservicios.
- Contenido concreto de nuevas librerías.
- Tags Nx y restricciones de dependencias.
- Límites estrictos entre aplicaciones y librerías.
- Separación Browser/Server, cuya aplicación específica a configuración se rige por ADR-002.
- Organización por capas o Clean Architecture.

Estas decisiones requieren requisitos propios y, cuando corresponda, ADR adicionales.

## 19. Dependencias

Dependencias técnicas actuales de la decisión:

- Nx como plataforma del workspace.
- pnpm como administrador de dependencias.
- TypeScript como lenguaje común.
- Plugins Nx para Next.js, NestJS/Node, Jest, Playwright, ESLint y Webpack.
- Configuración raíz mediante `nx.json`, `tsconfig.base.json` y `package.json`.
- Archivos `project.json` de aplicaciones y librerías.

La decisión no depende de que las integraciones externas estén implementadas.

## 20. Plan de implementación

### Implementación actual

1. Crear el workspace Nx.
2. Crear `customer-portal` y `admin-portal`.
3. Crear `api`.
4. Crear proyectos E2E asociados.
5. Configurar plugins y targets Nx.
6. Crear la librería compartida inicial `libs/config`.
7. Configurar el alias `@nexus/config`.

### Arquitectura propuesta e implementación futura

1. Definir tags por tipo, scope y plataforma.
2. Endurecer `@nx/enforce-module-boundaries`.
3. Completar la separación Browser/Server de configuración conforme a ADR-002.
4. Introducir librerías de capacidad solo junto con casos de uso aprobados.
5. Revisar el grafo Nx en cada cambio estructural.
6. Mantener documentación y ADR sincronizados.

La separación de configuración está aprobada por ADR-002 y permanece en implementación parcial. Los demás elementos son propuestas e implementación futura. Ninguno forma parte del alcance aprobado de este ADR.

## 21. Plan de validación

Para validar continuamente esta decisión se debe:

- Confirmar que Nx reconoce todas las aplicaciones y librerías.
- Verificar que los targets relevantes pueden resolverse por proyecto.
- Ejecutar build, lint y tests de proyectos afectados cuando un cambio lo requiera.
- Ejecutar E2E para los límites desplegables relacionados.
- Revisar que el grafo represente los proyectos incluidos en la decisión.
- Revisar el diff y la documentación en cambios estructurales.

No se ejecutan estas validaciones como parte de la redacción documental actual porque la tarea las prohíbe.

## 22. Plan de reversión

La adopción de Nx es una decisión estructural ya incorporada al repositorio. No debe revertirse mediante eliminación directa o cambios parciales.

Una reversión futura requeriría:

1. Crear y aceptar un nuevo ADR que reemplace este documento.
2. Definir la estructura alternativa y sus propietarios.
3. Inventariar proyectos, contratos, dependencias y pipelines.
4. Diseñar una migración incremental y reversible.
5. Preservar historial, pruebas y trazabilidad.
6. Validar builds, tests, despliegues y operación de cada unidad resultante.
7. Retirar configuración Nx únicamente después de completar la migración.

No se debe usar una reversión destructiva de Git ni dividir el repositorio sin autorización explícita.

## 23. Criterios de aceptación

La decisión se considera adoptada cuando:

- El repositorio contiene un workspace Nx válido.
- `customer-portal`, `admin-portal` y `api` son proyectos Nx independientes.
- Los proyectos E2E están asociados a sus aplicaciones.
- Las librerías compartidas se ubican bajo `libs/` y tienen punto de entrada público.
- Nx puede representar el grafo de proyectos.
- Los cambios pueden validarse mediante targets por proyecto o `affected`.
- La implementación actual y la arquitectura propuesta están documentadas por separado.

Los criterios estructurales principales están reflejados en el código versionado. Los tags, límites estrictos y demás evoluciones pertenecen a la arquitectura propuesta; no están aprobados por ADR-001 y no bloquean el estado aceptado de la decisión inicial. La separación Browser/Server específica de configuración se rige por ADR-002.
